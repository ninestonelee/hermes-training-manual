운영·확장 절차

VPS, 크론, 보안, 백업 운영

로컬 설치 후 Hermes를 24시간 운영하고, API 키·크론·GitHub 백업·여러 에이전트 분리를 안전하게 다루기 위한 운영 매뉴얼입니다.

대상로컬 검증을 끝낸 사용자
환경Linux VPS, SSH, systemd
결과상시 실행, 백업, 복구 가능한 운영

1. 사전조건

운영 환경은 로컬 설치보다 권한과 비용 리스크가 큽니다. 먼저 로컬에서 Hermes 기본 대화, provider 설정, 대시보드 확인을 마친 뒤 VPS로 옮깁니다.

  • 로컬 검증 완료: hermes, hermes doctor, provider 선택이 정상 동작해야 합니다.
  • VPS 접속 정보: Ubuntu 계열 Linux, SSH 접속, 방화벽을 관리할 수 있는 계정.
  • 비공개 백업 저장소: 메모리, 스킬, 운영 문서를 백업할 private GitHub 저장소.
  • 비밀키 원칙: 키는 채팅에 붙여넣지 않고 설정 파일 또는 전용 명령을 통해 저장합니다.
  • 비용 한도: provider별 월 한도, API 키 권한, 알림 설정을 먼저 정합니다.
!

자율 에이전트를 본인 PC보다 넓은 권한으로 운영하지 마세요. VPS, 전용 계정, 최소 권한 키를 사용해 실제 신입에게 권한을 주듯 단계적으로 열어야 합니다.

2. 설치

VPS 설치는 Linux 설치 절차와 같습니다. 한 대의 에이전트만 운영하면 직접 설치가 간단하고, 여러 에이전트를 분리하려면 컨테이너를 검토합니다.

직접 설치

VPS 사용자 계정에 바로 설치합니다. 처음 운영하거나 단일 에이전트만 사용할 때 가장 단순합니다.

컨테이너 분리

역할별 에이전트를 분리할 때 적합합니다. 키, 메모리, 도구 권한을 컨테이너 단위로 나눌 수 있습니다.

  1. SSH로 접속합니다. VPS 제공자가 안내한 사용자 계정으로 접속합니다.
  2. 공식 설치 명령을 실행합니다. Linux/macOS/WSL2와 같은 설치 명령을 사용합니다.
  3. 설정 마법사를 실행합니다. provider, 모델, 도구, gateway를 순서대로 설정합니다.
  4. 게이트웨이 서비스를 등록합니다. 메시징을 항상 받으려면 systemd 서비스로 띄웁니다.
VPS 설치
curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
hermes setup
게이트웨이 서비스
hermes gateway install
hermes gateway status

3. 설정

운영 설정의 핵심은 provider 비용, 비밀키 저장, 크론, 백업입니다. 모든 자동화는 “검증 가능한 작은 단위”로 쪼개 등록합니다.

Provider와 API 키

  • provider 선택은 hermes model에서 수행합니다.
  • Anthropic은 Max extra usage 또는 API 키 과금 조건을 확인해야 합니다.
  • Copilot, OpenAI, Nous Portal 등은 각 provider 문서에서 현재 인증 방식을 확인합니다.
  • 키를 입력해야 한다면 채팅창이 아니라 shell, 환경 파일, 전용 config 경로를 사용합니다.

크론 자동화

크론은 자연어로 등록하되 결과물, 실행 시간대, 실패 시 알림 대상을 명확히 씁니다. 서버 시간대가 UTC일 수 있으므로 “한국 시간 기준”처럼 기준을 적습니다.

크론 요청 예시
매일 한국 시간 오전 7시에 AI 뉴스 5개를 요약해줘.
출처 링크를 포함하고, 실패하면 게이트웨이 메시지로 알려줘.
이 작업을 크론으로 등록해줘.

GitHub 백업

메모리, 스킬, 운영 문서는 private 저장소에 백업합니다. .env, 토큰, 쿠키, provider credential 파일은 반드시 제외합니다.

백업 요청 예시
이 Hermes 운영 폴더를 private GitHub 저장소에 백업하는 절차를 만들어줘.
.env, 토큰, 쿠키, provider credential 파일은 제외하고,
매일 자정에 변경 요약을 커밋하도록 크론을 등록해줘.

4. 실행

운영 중에는 CLI, gateway, cron, 대시보드의 역할을 분리해서 사용합니다. 모든 작업을 하나의 긴 대화에 밀어 넣지 않는 것이 품질 유지의 핵심입니다.

작업명령 또는 위치용도
대화형 CLIhermes깊은 작업, 디버깅, 설정 변경
모델 변경hermes modelprovider와 기본 모델 선택
게이트웨이hermes gateway statusTelegram, Discord 등 메시징 상태 확인
크론hermes cron예약 작업 목록과 실행 상태 확인
대시보드http://localhost:9119세션, 작업, 스킬, 프로필 관리

텔레그램은 빠른 지시와 알림에 좋고, CLI는 장시간 코딩과 위험한 변경 검토에 좋습니다. 둘은 같은 에이전트를 바라보지만 가시성과 제어 수준이 다릅니다.

5. 검증

운영 검증은 “서비스가 떠 있다”보다 넓어야 합니다. 인증, 권한, 시간대, 백업 제외 규칙까지 확인합니다.

  1. 기본 진단hermes doctor가 통과하거나 조치 가능한 안내를 보여주는지 확인합니다.
  2. 게이트웨이 상태hermes gateway status에서 서비스가 실행 중인지 확인합니다.
  3. 예약 작업테스트용 짧은 크론을 하나 만들고 실행 결과가 오는지 확인합니다.
  4. 백업 제외저장소에 .env, 토큰, credential 파일이 올라가지 않는지 확인합니다.
  5. 복구 리허설새 폴더 또는 테스트 서버에서 백업 자료를 읽고 복원 절차가 이해되는지 확인합니다.
운영 검증 명령
hermes doctor
hermes gateway status
hermes cron

6. 장애 대응

증상우선 확인대응
gateway가 메시지를 받지 못함서비스 상태, 토큰, allowed usersgateway를 재시작하고 메시징 문서의 권한 설정을 확인합니다.
크론 시간이 어긋남서버 시간대, 요청 문구“한국 시간 기준”처럼 기준 시간대를 명시해 다시 등록합니다.
API 비용이 예상보다 큼provider, 모델, 자동화 빈도저렴한 모델로 전환하고 크론 주기와 최대 반복을 줄입니다.
메모리 때문에 이상 동작MEMORY.md, USER.md낡은 항목을 정리하고 변경 이유를 메모리에 반영합니다.
백업에 비밀이 포함됨최근 커밋, 히스토리즉시 키를 폐기하고 저장소 히스토리 정리 또는 새 저장소 전환을 검토합니다.

7. 복구

운영 복구는 “서버 재구축”, “인증 재설정”, “데이터 복원” 순서로 진행하면 혼란이 줄어듭니다.

  1. 서비스를 멈춥니다. 오작동 중인 gateway와 cron을 중지해 추가 변경을 막습니다.
  2. 비밀키를 보호합니다. 유출 가능성이 있으면 provider, GitHub, 메시징 토큰을 먼저 폐기합니다.
  3. 새 환경을 설치합니다. 공식 설치 명령으로 새 VPS 또는 사용자 계정에 Hermes를 설치합니다.
  4. 백업 자료를 복원합니다. 메모리, 스킬, 운영 문서를 가져오되 .env는 새 값으로 다시 만듭니다.
  5. 작은 작업으로 재검증합니다. provider 응답, gateway 수신, 테스트 크론, 백업 커밋을 순서대로 확인합니다.
!

OpenClaw 등 다른 에이전트에서 옮길 때는 마이그레이션 명령이 현재 버전에 존재하는지 hermes --help로 먼저 확인하세요. 문서나 영상의 명령은 버전 차이가 날 수 있습니다.

8. 수동 검증 체크리스트

  • VPS 설치 후 hermes doctor와 hermes gateway status를 확인했다.
  • provider별 인증과 비용 조건을 공식 문서 또는 설정 화면에서 확인했다.
  • API 키, 토큰, credential 파일이 채팅·문서·Git 커밋에 노출되지 않는다.
  • 테스트 크론을 등록해 실제 실행 결과와 시간대를 확인했다.
  • private 백업 저장소에 .env와 비밀 파일이 제외되는지 확인했다.
  • 모바일/데스크톱에서 공통 내비게이션과 문서 목차를 키보드로 사용할 수 있다.
  • 깨진 내부 앵커와 죽은 로컬 링크가 없다.
